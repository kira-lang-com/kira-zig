description: VM, LLVM/native, hybrid, and the runtime/native boundary used by current Kira programs.

Kira currently has three execution targets.

| Target | What runs | Best for |
| --- | --- | --- |
| VM | bytecode under `kira_vm_runtime` | simplest runnable subset and fast iteration |
| LLVM/native | native object + host executable | fully native proofs and direct native execution |
| hybrid | bytecode plus native shared library | mixed `@Runtime` and `@Native` programs, especially FFI-heavy ones |

## VM

The default `run` path is the VM:

```bash
kira run examples/hello
```

The pipeline is:

```text
source -> lexer -> parser -> semantics -> IR -> bytecode -> VM
```

VM mode is the simplest backend, but it cannot execute `@Native` functions. If you try, the build layer emits `KBUILD001`.

## LLVM / Native

The native path lowers the shared IR through the LLVM C API and links a host executable:

```bash
kira run --backend llvm examples/callbacks
```

The pipeline is:

```text
source -> lexer -> parser -> semantics -> IR -> LLVM IR -> object file -> native executable
```

This is the right target when the whole program should run natively and the entrypoint itself is native-capable.

## Hybrid

Hybrid mode is where Kira's mixed execution model shows up directly:

- `@Runtime` functions stay in bytecode
- `@Native` functions compile into a native shared library
- a hybrid manifest ties the module together
- `kira_hybrid_runtime` loads the bytecode sidecar and the native library in one process

Example:

```bash
kira run --backend hybrid examples/hybrid_roundtrip
```

Current output:

```text
native main
runtime helper
native helper
```

## Why Hybrid Is The Main Recommended Path

For programs that need both:

- Kira-managed runtime code, and
- real native work such as FFI calls or native callbacks

hybrid is the most natural path in the repo today.

It lets you keep:

- runtime-oriented logic under `@Runtime`
- native-only work under `@Native`
- a single mixed module boundary instead of forcing everything into VM-only or native-only shape

Pure LLVM/native is still useful, especially for all-native proof targets like the callbacks and Sokol triangle examples. VM remains the simplest default when you do not need native execution.

## Cross-Boundary Execution

The hybrid boundary is explicit.

Runtime-to-native:

- the VM emits `call_native`
- `kira_native_bridge` resolves a native trampoline from the shared library
- arguments and results are marshalled through bridge values

Native-to-runtime:

- the native bridge installs a runtime invoker callback
- generated native wrappers call back into the runtime with function ids and bridge values

The current bridge value set covers:

- `void`
- integer
- string
- boolean
- raw pointer

That is the concrete contract behind the hybrid examples and FFI callback tests.
